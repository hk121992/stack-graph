## 1. Problem & motivation

The incremental arc (`triage → specify-slice → build → review → land`) exists and traverses
end-to-end. What does not exist is the thing that runs a **batch** of standalone IUs well. The
first production run (be-civic harness, 6 IUs) ran them sequentially in one session and proved
both halves of the case:

- **It works** — 4 built + fresh-context-reviewed, 1 correctly escalated at specify-slice, 1
  correctly handled as a route-out.
- **It degrades** — context was heavy by IU 4. This is exactly the failure mode `build`'s own
  doctrine names ("a long context degrades; an agent's best work happens in a fresh window"),
  currently scoped to a work-item's child IUs *within* one build span. The same principle, lifted
  to the arc level, says: **one standalone IU = one fresh agent session.**
- **The contract already supports it** — one IU of the batch ran in a separate cold session off
  the carrier file alone (chip-spawned) and succeeded. `goal / files / acceptance /
  acceptance_check / verification` on the carrier-lite are sufficient context for a cold
  traversal. Event-log keying by `(carrier_id, carrier_kind, arc)` means interleaved sessions
  cannot bleed stage projections. The three-writers split means a dispatcher never needs to
  write lifecycle — it only observes outcomes.

Two hard lessons from the run are **requirements, not preferences**:

1. A concurrent session in a **shared checkout** switched branches mid-flight and a commit landed
   on the wrong branch (recovered via temp worktree + `reset --keep`). Worktree isolation per
   dispatched IU is mandatory.
2. IUs in one batch touched **three repos**; one real merge conflict occurred between a slice
   branch and a commit that landed on `dev` mid-batch. Integration must be repo-aware and must
   re-prove each slice's `acceptance_check` on the merged tree.

## 2. What loop-runner is — and is not

`loop-runner` is an **orchestration node** (primitive: skill) that takes the operator-approved
subset of open standalone IUs — each already **decision-complete from raise time** (§10: `triage`
captures the raising conversation's context and asks every operator question on the spot) — and
dispatches **one fresh agent session per IU**, each running `specify-slice (unattended) → build →
review` against the carrier file in isolated worktrees. It owns **IU selection mechanics, the
decision-completeness gate, isolation, concurrency, dependency order, route-out handling, and the
batch report + land queue**. It is the arc-level twin of what `build` already does for a
work-item's child IUs within one span.

> **History of this seam (rounds 1–3):** issue #19 sketched the dispatched session running
> `specify-slice → build → in-span review`. Review round 1 (H1) found that contradicts
> specify-slice's collaborative contract and moved specification to attended intake. The operator
> then redesigned the front (§10): the contradiction is resolved **upstream instead** — all
> operator questions move to **raise time at `triage`** (which captures the live conversation's
> context into the carrier), and `specify-slice` is **lightened** to a formalise-and-enforce pass
> with an **unattended mode** (the same body-mode pattern as `review`'s headless), so it returns
> to the dispatched session with nothing left to ask. The field evidence supports it: the
> chip-spawned cold session ran specify → build off the carrier alone and succeeded.

It is **not**:

- **A stage of the incremental arc.** The arc's stages are unchanged. `loop-runner` declares no
  `composes-into` and no process edges (`precedes`/`can-follow`) — it is an
  **orchestrator-over-traversals**: it dispatches sessions that traverse the arc; it never
  traverses the arc itself. The stage nodes inside each dispatched session emit the carrier-keyed
  enter/exit events; the projection sees an ordinary incremental traversal per carrier.
- **A gate-holder.** Operator gates stay outside it on both ends. Input: the operator approves
  the batch subset (`proposed` IUs are suggestions, per triage — never "everything in the
  manifest"). Output: a land-decision queue; the single `commit-to-land` gate per IU is fired by
  `land`, which loop-runner invokes per IU only after the operator's decision. **The dispatcher
  merges nothing on its own.**
- **A retry loop.** Route-outs (`review-flagged`, `escalated`, `hitl-parked`, `blocked`) park
  with a reason and a pointer; the dispatcher never re-dispatches a routed-out IU within the
  batch.
- **A lifecycle or shared-surface writer during the span.** Per the three-writers split it writes
  no lifecycle: `in-delivery` is set by the build-enter event inside the dispatched session; the
  terminal state is the land gate's. The **improvements-manifest** — already specified as a
  derived index (directory-topology) — is regenerated by the coordinator (refresh-index) after
  outcomes return; **dispatched sessions never write it**, and never write any shared committed
  surface (see §4, escalation enactment).

## 3. Position in the graph

```yaml
id: loop-runner
primitive: skill
mode: collaborative          # collaborative bookends (intake + land queue);
determinism: generative      # autonomous dispatch span between them — same shape as build
edges:
  invokes:
    - { id: triage }         # the intake define-now fallback — its capture flow + cold-handoff
                             # gate, run attended on an under-defined stub (review R3-M3)
    - { id: specify-slice }  # the dispatched session's ENTRY stage (unattended mode)
    - { id: build }          # reached inside the session via specify-slice's own precedes edge;
                             # also a direct entry when an IU is already formalised
    - { id: land }           # invoked per IU at the close phase, after the operator's decision
  references:
    - { id: IU-schema,                load: import }
    - { id: carrier-interface,        load: on-demand }
    - { id: instrumentation-preamble, load: import }
    - { id: bindings-contract,        load: on-demand }   # improvements-root/-manifest, event-log
    - { id: deploy-config,            load: on-demand }   # per-repo roots + branch policy (the
                                                          # same surface ship/deploy already read)
```

Notes on the edge choices:

- **`invokes` to arc stage nodes is structurally legal** (02-graph-spec permits `invokes:
  node → node`; nothing restricts the target class) but **no existing node invokes a stage**, so
  this is argued on its own merits, not by analogy: the dispatcher needs exactly four reach-ins —
  run `triage`'s capture flow attended on an under-defined stub at intake (the define-now
  fallback), start a fresh session at `specify-slice` (unattended mode), enter at `build` directly
  when an IU is already formalised, and fire `land` per queue decision. Each is a genuine invocation from the coordinator's context (or the
  dispatch boundary); none is arc sequence. The 07-decomposition amendment
  (orchestrator-over-traversals) carries the doctrine. The targets declare no inbound edge
  (caller-side convention).
- **No `invokes: review`.** The session reaches review via the arc's own `precedes` chain
  (`specify-slice → build → review`) *inside* the traversal — declaring it here would misstate
  who reaches it.
- **No `composes-into incremental`.** The dispatched sessions traverse the arc; the dispatcher
  does not. The carrier-keyed events the projection reads come from the stage nodes inside the
  sessions, exactly as in a hand-run traversal. (This is the 02-graph-spec clarification: a
  traversal is carrier-scoped, not session-scoped.)
- **No process edge to `triage`.** Triage's relationship to loop-runner is dataflow (it scaffolds
  the `proposed` IUs the operator later approves into a batch), not arc sequence. Prose, not an
  edge — consistent with "not everything is a node/edge."

## 4. The dispatch protocol

### Phase 0 — Intake (collaborative; the operator is present throughout)

1. Resolve `improvements-root` / `improvements-manifest` via the bindings. Read the manifest;
   list open standalone IUs (`lifecycle_state: proposed` or `in-delivery`, not terminal).
2. **The operator approves the batch subset.** Present the candidates (id, title, improves,
   slice_type if tagged, size, files); the operator picks. Never auto-select the whole manifest.
3. **Decision-completeness gate (the readiness check).** Per approved IU, verify the carrier is
   **decision-complete from raise time** (§10): the body carries `## Context` (verbatim evidence
   — observed vs expected, repro, pointers) and `## Decisions` (every call settled, including the
   AFK/HITL call and any inter-IU dependency), and no unresolved fork remains in prose — the
   IU-schema loop-eligibility invariant. An under-defined stub does **not** dispatch: the fallback
   is **define-now attended** — invoke **`triage`'s capture flow** (harvest + cold-handoff gate;
   the operator is present at intake anyway) — or park.
4. **Pre-park what cannot run unattended.** Read the AFK/HITL call from the carrier's recorded
   decisions: a HITL slice whose pause stage is in-session (`specify-slice`, `build`, `review`)
   is parked to the attended queue — its pause cannot be honoured unattended. A pause at `land`
   is dispatchable — land's gate is operator-held at the close phase anyway. (`slice_type` is
   *formalised* by specify-slice in-session; the *decision* is readable at intake from the
   Decisions record.)
5. **Freeze the git baseline.** Map each IU's `files` → repo(s) **via the harness's
   `deploy-config` surface** (per-repo roots + branch policy — the same surface `ship`/`deploy`
   already read; no new binding key). Capture a **frozen base-ref per repo** for the batch; every
   independent IU branch in that repo is cut from it (no live-HEAD drift between dispatches). An
   IU touching a repo absent from the config, not checked out, or not writable is parked
   `blocked` now.
6. **Build the schedule:** dependency order first (a standalone IU's `dependencies` may reference
   other standalone IUs; for a stub the dependency is a raise-time **decision** recorded in
   `## Decisions` — the schedule reads the formalised field where present, the recorded decision
   otherwise, R3-M2). A dependent is not dispatched until its dependency's session returns
   `built`, and its branch is then **cut from the dependency branch's tip** (stacked), not from
   the bare base-ref — a dependent must build against its dependency's committed output. A
   dependency **cycle**, or a dependency id missing from the batch and not already landed, parks
   both ends `blocked` at intake (malformed). **Dependency exception to the no-shared-scope rule
   (R3-H1):** scope overlap explained by a declared dependency edge is expected — the dependent
   consumes the dependency's output and the stacked cut already orders them; the overlap rule
   serialises only IUs with no dependency relationship. **Dependency cascade rule:** if a dependency
   returns any non-`built` outcome, its dependents are **transitively parked** `blocked` (reason:
   dependency `<id>` returned `<outcome>`), never dispatched, their slots freed. Then the
   **no-shared-scope rule, late-binding** — `files` is formalised in-session by specify-slice, so
   at intake serialise on **`improves`-target overlap + the Context section's declared scope**
   (the provisional map); then **late-bind**: specify-slice's carrier-keyed stage-exit event
   carries the formalised `files`, and the coordinator re-checks overlap against in-flight sets
   before each subsequent dispatch (build's Parallel Safety Check, lifted to the batch level).
   Worktree isolation makes an undetected overlap a merge-conflict-at-dry-run problem, not
   corruption — the Phase 2 dry-run is the backstop. An already-formalised IU uses its real
   `files` directly. Concurrency cap: a dial, default **3** in-flight sessions; the harness
   overlay may tune it.

### Phase 1 — Dispatch span (autonomous)

Per scheduled IU:

1. **Create an isolated worktree per repo the IU touches**, branch `iu/<id>` cut from the frozen
   base-ref — native `isolation: 'worktree'` where available, script fallback otherwise (build's
   mechanism, reused). **Mandatory, no shared-checkout dispatch ever** — requirement #1 from the
   field, not an optimisation. Guard: if `iu/<id>` already exists (a cross-batch re-dispatch),
   surface it — reuse or recreate is an explicit choice, never a silent overwrite.
2. **Dispatch one fresh agent session** with the spawn bundle: the carrier file path (the
   decision-complete carrier is sufficient — proven), **entry stage `specify-slice` in unattended
   mode** (entry `build` when the IU is already formalised), the worktree path(s) and branch, the
   event-log binding, and the return-envelope contract. The session runs `specify-slice → build →
   review` per the arc's own node bodies — specify-slice **formalises** the captured definition
   into the content fields and enforces the invariants (routing out on a gap, never asking);
   build runs the tracer-bullet inner loop; review runs `headless`/`autofix` for an AFK slice —
   honouring the arc's escalate and HITL semantics, and **stopping after the review verdict — it
   never runs `land`.**
   **Write discipline inside the session:** the session writes its worktree(s) and **its own
   carrier file only** (one session per IU ⇒ single-writer, race-free); it never writes the
   manifest or any other shared surface. **Event appends** follow the append contract: one event
   = one whole line written in a single `O_APPEND` write (shell `>>` of one line) — never
   buffered multi-write — so concurrent sessions cannot interleave partial records.
3. **Collect the return envelope:**
   `{ outcome: built | review-flagged | escalated | hitl-parked | blocked, commits[], branch(es),
   evidence (raw acceptance_check output + review verdict), tokens }`.
   - `built` — slice committed on its branch, acceptance evidence attached, review verdict
     clean or all-deferred; joins the land queue.
   - `review-flagged` — the session's `review` pass ended with **unresolved actionable findings**
     (`gated_auto`/`manual`, which headless/autofix modes may not apply). In-session the arc's
     review → build fix loop runs **bounded** (max 2 re-entries per the cyclic-edge discipline);
     a slice still flagged after that returns here with the ranked finding set and joins the
     **attended queue**, not the land queue — the operator triages the findings.
   - `escalated` — a mid-build promote signal (a hidden fork or decomposition surfaced after
     intake). The session **stops and returns the signal + rationale; it does not enact**
     promotion (that mutates shared surfaces). The **coordinator enacts attended** at the close
     phase: the arc's normal escalate path — work-item created/reused via
     `product-dashboard-curator add-item`, the IU closed `dropped` (reason: promoted), two-way
     provenance recorded. (The `dropped`-via-promotion write is **already legal** — it is the
     `escalates` edge's spec'd behaviour in 02-graph-spec, not a new writer; the coordinator
     merely enacts it attended instead of the isolated session.)
   - `hitl-parked` — the session hit a pause it could not honour unattended; parked for an
     attended run, with the decision point surfaced. (Rare post-intake, since the AFK/HITL call
     is recorded at raise and read at the intake pre-park — this catches discoveries, not known
     tags.)
   - `blocked` — scope/dependency/environment blocker, an **insufficiently-defined carrier**
     specify-slice could not formalise without asking (reason recorded; back to raise-capture),
     or a dead session (crash/timeout, with the failure evidence). Parked with the reason. Never
     retried in-batch.
   Route-outs **write no lifecycle**: the IU stays open and re-lists in a future batch; only the
   operator parks it durably (the land gate's `parked` decision) or the coordinator's enacted
   promotion drops it.
4. **Emit the carrier-keyed `dispatch-complete` event** — the same hook-captured
   subagent-completion event class as build's `unit-complete` (D57 mechanism), carrying the
   triple `(carrier_id, carrier_kind, arc)`, the outcome, and **`tokens_per_session`** — the
   whole-session cost of that IU's traversal: a defined **superset of** the `tokens_per_iu`
   build emits inside the session (the specify, build, and review spans + session overhead). Build's
   `tokens_per_iu` remains canonical for the IU-sizing dial; `tokens_per_session` is the
   **dispatch-efficiency** measure (sequential baseline: ~35–80k output tokens per IU, rising
   with session length — the comparison this node exists to win). `dispatch-complete` is **not a
   stage event**: the projection never reads it for `current_stage`.

### Phase 2 — Integration dry-run + land queue (collaborative)

1. **Batch report.** Per-IU outcomes with evidence; the five buckets named; escalations enacted
   attended (per Phase 1.3); manifest regenerated (refresh-index) from the carrier files.
2. **Integration dry-run per repo** (requirement #4): in a **scratch worktree**, merge the
   `built` branches onto the frozen base-ref *plus current target-branch state* in dependency
   order and **re-run each slice's `acceptance_check` on the merged tree** — cross-slice and
   mid-batch-drift conflicts surface here, not in prod. The scratch tree is then discarded: this
   step produces **readiness evidence**, it performs no real merge. A merged-tree failure pauses
   that repo's queue and surfaces options (fix in-place if trivial, re-dispatch a fix session
   against the same carrier, park) — operator decides.
3. **Land queue.** Per `built` IU, surface the land decision with the dry-run evidence; on the
   operator's go, **invoke `land`** for that IU — land fires its own single `commit-to-land`
   gate (writing the terminal `lifecycle_state` + `gate_decisions` entry) and **owns the real
   merge** via its ship → deploy sub-arc, per the consuming repo's branch policy (two-track
   repo: PR → merge to `dev`, review at rollup — land's non-deploying/staging-only shape;
   single-branch repo: the push *is* the land gate). One merge owner, no double-merge: the
   dispatcher's Phase 2.2 merge is a discarded dry-run; land's is the real one. Land's existing
   dial applies: an AFK slice in a mature harness may auto-record on green; HITL raises a hard
   gate.
4. **Worktree teardown — merged branches only.** For a landed IU, build's order-bearing sequence
   applies (test on merged result → unlock → remove → `git branch -d`, never `-D`). For a
   **parked outcome** (`review-flagged` / `hitl-parked` / `blocked`), the worktree is removed but
   the **branch is retained** with its name recorded in the batch report — `git branch -d` would
   rightly refuse the unmerged work, and a parked slice's commits are the resume point for the
   attended follow-up. Retained `iu/<id>` branches are why Phase 1.1 carries the branch-exists
   guard.

## 5. Instrumentation summary

- Dispatcher node-enter / node-exit events (imported preamble) carry the batch id and are
  **non-carrier events** — routed to the **factory-conformance stream**, not product-outcomes,
  so they never touch the carrier projection.
- Per-IU **dispatch-complete** event (product-outcomes): carrier triple + outcome +
  `tokens_per_session` (defined in §4 Phase 1.4; 06-analytics amendment).
- All stage-level events (enter/exit, unit-complete with `tokens_per_iu`, gates) are emitted by
  the stage nodes **inside** the dispatched sessions, unchanged — concurrent sessions append to
  the same `event-log` under the **append contract** (one event = one whole line in a single
  `O_APPEND` write; part of the spawn bundle and a sentence in the 06-analytics amendment —
  the preamble is method-agnostic today, so the discipline must be stated, not assumed).
- **Specify-slice's stage-exit event doubles as the scheduling signal**: it carries the
  formalised `files`, which the coordinator reads to late-bind the no-shared-scope check before
  each subsequent dispatch (Phase 0.6).
- **The projection triple-key is now load-bearing and un-gates with this node.** The
  `(carrier_id, carrier_kind, arc)` key separates interleaved carriers *in the model*; the
  implementation in `publish-projection.ts` was input-gated on incremental-arc events existing —
  they now do (be-civic, 2026-06-10), and concurrent dispatch makes the key mandatory, not
  cosmetic. It joins this build's scope (§7.3). dispatch-complete is a measure event, never a
  stage event — the projection never reads it for `current_stage`.

## 6. Decisions taken (builder's call per the issue), flagged not blocking

1. **Dispatch mechanism: mechanism-agnostic contract, native subagents default.** The node body
   specifies the spawn-bundle / return-envelope contract and names two conforming mechanisms:
   native subagent dispatch with `isolation: 'worktree'` (default where the runtime offers it)
   and a headless session (`claude -p`) fallback. The contract, not the mechanism, is canonical —
   keeps the node general across harnesses and runtime versions.
2. **All operator questions move to raise time at `triage`; specify-slice runs unattended in the
   dispatched session** (operator redesign, §10 — supersedes round 1's attended-intake
   resolution).
3. **The dispatched session runs the real `review` stage** (headless/autofix for AFK), not a
   bespoke "in-span review" — arc conformance stays observable, and build's own in-span reviewer
   remains the recommended pre-check it already is.
4. **Concurrency default 3**, harness-tunable. Conservative: the no-shared-files rule plus
   dependency order dominate; the cap mostly bounds host load.
5. **`tokens_per_session` kept as a distinct measure** — but now explicitly defined as a superset
   of build's `tokens_per_iu` (which stays canonical for sizing), resolving the double-emit. The
   issue's literal wording was "emit `tokens_per_iu` per session"; on the incremental arc one
   session ≈ one IU, so the two nearly coincide — the distinct name preserves the build-span vs
   whole-session distinction that the dispatched-vs-sequential comparison needs.

1. **The node** — `graph/loop-runner/` (research-report + canonical `loop-runner.md`) per
   `02-graph-spec`, edges as §3, body as §4–5. Authored via `sg-graph-maintainer new`.
2. **Spec amendments** — the three rows marked "amendment" in the touchpoints table
   (02-graph-spec one sentence; 07-decomposition the orchestrator-over-traversals pattern;
   06-analytics the dispatch-complete event class + `tokens_per_session` definition + the
   conformance-stream routing of the dispatcher's own events).
3. **Workspace backend (un-gated by this node — the inputs now exist):**
   `publish-projection.ts` keyed by the `(carrier_id, carrier_kind, arc)` triple, and the
   publisher/measure-outcomes allowlists extended to admit the `dispatch-complete` event and
   `tokens_per_session` — without this the node's calibration measure is silently dropped
   (Codex R2-M6). Hostile-fixture coverage per the existing publisher test pattern.
4. **Record refresh** — `sg-graph-maintainer index` (graph-record.json) + validate.
5. **Out of scope, named:** the Workflow-script prototype lives harness-side (the issue notes a
   harness can prototype the shape today); plugin vendor bump is the standard post-merge
   `chore(vendor)` step.

## 10. Operator redesign (2026-06-10) — raise-time capture; the front re-split

After rounds 1–2 the operator redirected the arc front. Direction (three constraints): an IU
marked incremental only runs in the loop **when all relevant decisions have already been made**;
that happens **at IU authoring time** ("I say raise the IU and the agent asks me any questions to
define it on the spot"); the loop scope is **technical / bug-shaped** work. Most raises happen in
a session that **already holds the context** — so definition is a *capture*, not an interview,
and the additional questions should be limited; the skill must nonetheless **ensure the context
is complete**. The operator explicitly kept both front nodes: **reinforce `triage`, lighten
`specify-slice`** (a proposed fuse-and-retire variant was rejected).

The resulting split is **human-facing vs mechanical**, not route-vs-define:

| | `triage` (reinforced) — capture & decide, with the operator | `specify-slice` (lightened) — formalise & enforce, without |
|---|---|---|
| Trigger | the operator phrase **"raise an IU"** (added to description/when-to-use) | reached as the dispatched session's entry (unattended mode), or hand-run (collaborative default) |
| Does | route rule (+ technical/bug-shaped tie-breaker: specification derivable from carrier + codebase, no product/design judgment ⇒ incremental); **harvest** the live conversation into the carrier body — `## Context` (observed vs expected, repro, error output, pointers — verbatim evidence, never "see this conversation") + `## Decisions` (every call settled, incl. the AFK/HITL call); **gap-check** against the cold-handoff test and ask the remaining questions **now** (typically 0–2, because the context is harvested) | **render** the captured definition into the formal content fields (`goal/files/acceptance/acceptance_check/size/slice_type/verification`), using `explore` for mechanical detail; **enforce** the vertical-slice / testing / single-slice invariants; formalise `slice_type` from the recorded decision |
| Never | hands off a carrier that fails the cold-handoff test — an unresolved fork is settled here, escalated, or the IU is not raised (new invariant) | asks the operator in loop mode — a gap or late-surfacing fork is a **route-out** (`blocked: insufficiently-defined` back to raise-capture, or `escalated`), never a question |
| Writes | identity frontmatter (as today) + the **definition body** (Context + Decisions) | the content fields (as today), in-session, on its own carrier (single-writer) |

**The cold-handoff test** (the completeness contract, homed in IU-schema as the loop-eligibility
invariant): *a fresh agent with only this carrier file and repo access could implement and prove
the slice* — field-proven by the chip-spawned cold session. Checklist: outcome-framed intent in
Context; evidence verbatim; scope pointers real; no unresolved decision in prose. A `proposed`
stub without a passing definition body is a valid intra-conversation state but **never
loop-eligible**.

**Knock-ons applied in this revision:** loop-runner intake Phase 0.3 = the decision-completeness
gate (define-now attended fallback); pre-park reads the recorded AFK/HITL decision; dispatched
entry returns to `specify-slice (unattended) → build → review`; scheduling goes late-binding
(improves-overlap at intake, formalised `files` off specify-slice's stage-exit event, dry-run as
backstop); `tokens_per_session` spans specify+build+review; `blocked` gains the
insufficiently-defined reason. Handbook touch-ups: 07-decomposition worked-case span and
06-analytics tokens_per_session span. Question-shape discipline baked into triage: **propose,
don't interrogate** — derive what the repo can answer, ask only genuine decisions.
